---
title: "Secrets management across all Xinity deployment types"
description: "Xinity supports the _FILE secrets convention across all deployments. Covers systemd LoadCredential, Docker secrets, NixOS options, and rotation steps."
---

All three Xinity deployment methods — the CLI (systemd), Docker Compose, and NixOS — share the same `_FILE` convention for injecting secrets. For any environment variable `KEY`, you can set `KEY_FILE` to a file path instead of providing the value directly. The service reads the file at startup and uses its trimmed contents as the value. Direct environment variable values take precedence over their `_FILE` counterparts.

## Which variables are secrets

Variables are designated as secrets by annotating their Zod schema entries with `.meta(secret())` in the source. The variables currently marked as secrets are:

| Variable | Services |
|----------|----------|
| `DB_CONNECTION_URL` | Gateway, Dashboard, Daemon |
| `REDIS_URL` | Gateway |
| `BETTER_AUTH_SECRET` | Dashboard |

These are the values that should never appear in plain text in a process environment or a world-readable file.

## CLI / systemd deployment

The Xinity CLI automatically separates config from secrets during `xinity up`. It inspects each component's schema and stores values in two locations:

| Type | Location | Permissions | Loaded via |
|------|----------|-------------|------------|
| Config | `/etc/xinity-ai/<component>.env` | `644` (readable) | systemd `EnvironmentFile=` |
| Secrets | `/etc/xinity-ai/secrets/<KEY>` | `600` (root only) | systemd `LoadCredential=` |

At runtime, systemd loads each secret file into an ephemeral credentials directory at `/run/credentials/xinity-ai-<component>/`. The service receives a `KEY_FILE` environment variable pointing to that path and reads the secret from the file. **The secret never appears as a plain environment variable in the process environment.**

### File layout after `xinity up gateway`

```
/etc/xinity-ai/
  gateway.env                    # HOST=0.0.0.0, PORT=4121, ...  (mode 644)
  secrets/                       # (mode 700)
    DB_CONNECTION_URL            # postgresql://...               (mode 600)
    REDIS_URL                    # redis://...                    (mode 600)
```

### Rotating secrets (CLI)

```bash
# Write the new secret value
printf '%s' 'new-value' | sudo tee /etc/xinity-ai/secrets/KEY > /dev/null

# Restart the affected service
sudo systemctl restart xinity-ai-<component>
```

Or re-run the interactive prompts:

```bash
xinity configure <component>
```

## Docker Compose deployment

Two options are available, with Docker secrets recommended for production.

<Tabs>
  <Tab title="Option A: .env file">
    Store all secrets in the `.env` file alongside `docker-compose.yml`:

    ```bash
    chmod 600 .env
    ```

    This is straightforward for single-server deployments. The trade-off is that secrets end up as container environment variables and are visible via `docker inspect`.
  </Tab>
  <Tab title="Option B: Docker secrets (recommended)">
    Use the Docker `secrets:` block to mount secrets as in-memory files. The service reads them via `_FILE` environment variables, and they are never exposed as environment variables inside the container.

    **Step 1 — Create secret files:**

    ```bash
    mkdir -p secrets
    openssl rand -base64 32 > secrets/postgres_password
    openssl rand -base64 32 > secrets/redis_password
    openssl rand -base64 32 > secrets/better_auth_secret
    chmod 600 secrets/*
    ```

    **Step 2 — Reference them in a compose override:**

    ```yaml
    # docker-compose.override.yml
    services:
      gateway:
        environment:
          DB_CONNECTION_URL_FILE: /run/secrets/db_connection_url
        secrets:
          - db_connection_url
          - redis_url

      dashboard:
        environment:
          DB_CONNECTION_URL_FILE: /run/secrets/db_connection_url
          BETTER_AUTH_SECRET_FILE: /run/secrets/better_auth_secret
        secrets:
          - db_connection_url
          - better_auth_secret

    secrets:
      db_connection_url:
        file: ./secrets/db_connection_url
      redis_url:
        file: ./secrets/redis_url
      better_auth_secret:
        file: ./secrets/better_auth_secret
    ```

    With this configuration, secrets are mounted as in-memory files at `/run/secrets/` inside each container.
  </Tab>
</Tabs>

### Rotating secrets (Docker)

1. Update the secret file in `secrets/`:
   ```bash
   printf '%s' 'new-value' > secrets/KEY
   chmod 600 secrets/KEY
   ```
2. Restart the affected services:
   ```bash
   docker compose restart gateway dashboard
   ```

## NixOS deployment

The NixOS module supports three approaches for providing secrets:

<AccordionGroup>
  <Accordion title="Direct values in Nix config">
    Pass secret values directly in the module configuration. Simple but stores secrets in the Nix store, which is world-readable.

    ```nix
    services.xinity-gateway = {
      enable = true;
      dbConnectionUrl = "postgresql://user:password@host/db";
    };
    ```
  </Accordion>
  <Accordion title="Environment file">
    Point the service at a file containing environment variable assignments. The file is loaded at service start and is not part of the Nix store.

    ```nix
    services.xinity-gateway = {
      enable = true;
      environmentFile = "/etc/xinity-ai/gateway.secrets";
    };
    ```

    ```bash
    # /etc/xinity-ai/gateway.secrets (mode 600)
    DB_CONNECTION_URL=postgresql://user:password@host/db
    REDIS_URL=redis://:password@localhost:6379
    ```
  </Accordion>
  <Accordion title="Per-secret *File options">
    Use the per-secret `*File` options in the module, which map directly to the `_FILE` convention. The service reads each file at startup.

    ```nix
    services.xinity-gateway = {
      enable = true;
      dbConnectionUrlFile = "/run/secrets/db_connection_url";
      redisUrlFile = "/run/secrets/redis_url";
    };
    ```

    This is the recommended approach when using a secrets manager such as [sops-nix](https://github.com/Mic92/sops-nix) or [agenix](https://github.com/ryantm/agenix), which can provision secret files at the correct paths with appropriate permissions.
  </Accordion>
</AccordionGroup>

## Summary

<Info>
  Regardless of deployment method, the `_FILE` convention works the same way: set `KEY_FILE` to a path, and the service reads that file instead of looking for `KEY` in the environment. Direct values always take precedence over `_FILE` variants.
</Info>
